# 프로젝트 지시서: Stamp-a-Day (하루 한 장 우표 일기 앱)

## 1. 프로젝트 개요

매일 사진 한 장을 "우표" 형태로 만들어 월별 그리드에 붙여가는 비주얼 일기 앱.
컨셉 레퍼런스: 실물 "Punch Stamp Notebook" — 하늘색 배경에 빈 우표 칸(톱니 테두리)이 그리드로 깔려 있고, 사용자가 하루에 한 칸씩 채워나가는 경험.

- 플랫폼: iOS + Android (Expo / React Native)
- 출시 전략: MVP 완성 후 반응 보고 정식 출시 여부 결정 (TestFlight/내부 배포 우선)
- 핵심 감성: 아날로그 우표 수집 노트의 디지털화. 채워지는 재미, 빈 칸이 주는 동기부여

## 2. v1 범위 (이번에 만들 것)

**포함:**
- 로컬 저장 기반 개인 일기 (로그인 불필요, 설치 즉시 사용)
- 월별 스탬프 보드 (메인 화면)
- 사진 → 우표 만들기 플로우 (촬영/앨범 → 크롭 → 톱니 프레임 → 붙이기)
- 스탬프 상세 보기 / 메모 / 수정 / 삭제
- 매일 리마인더 로컬 알림
- 과거 날짜 채우기 (backfill)

**제외하되 구조적으로 대비할 것 (v2/v3 예정):**
- v2: 클라우드 백업 + 계정 (Supabase Auth/Storage 예정)
- v3: 공유/피드 기능 (다른 사용자 페이지 구경)
- 따라서 데이터 모델과 저장 계층은 추후 서버 동기화를 전제로 설계할 것 (아래 5번 참조)

## 3. 핵심 기능 명세

### 3.1 월별 스탬프 보드 (메인)
- 해당 월의 일수만큼 우표 칸을 그리드로 표시 (5열 권장, 세로 스크롤)
- 빈 칸: 회색 placeholder + 톱니 테두리 + 날짜 숫자 옅게 표시
- 채워진 칸: 사진 우표 썸네일 + 날짜
- 오늘 칸: 시각적 강조 (점선 테두리 또는 은은한 하이라이트)
- 좌우 스와이프 또는 헤더 화살표로 월 이동
- 빈 칸 탭 → 스탬프 만들기 플로우 진입 (해당 날짜로)
- 채워진 칸 탭 → 상세 뷰
- 미래 날짜 칸: 비활성 (탭 불가, 더 옅게 표시)

### 3.2 스탬프 만들기 플로우
1. 사진 소스 선택: 카메라 촬영 or 앨범
2. 크롭: 우표 비율 4:5 (1" × 0.8" 노트 기준) 고정 크롭
3. 미리보기: 톱니 프레임이 씌워진 최종 모습 + 한 줄 메모 입력(선택)
4. "붙이기" 버튼 → 우표가 그리드 칸에 착 붙는 짧은 애니메이션 (스케일+페이드, 200~300ms)

규칙:
- 하루 딱 1장. 이미 채워진 날짜는 새로 만들기 불가 (수정/교체는 상세 뷰에서)
- 과거 날짜는 빈 칸이면 언제든 채우기 가능
- 미래 날짜는 불가

### 3.3 상세 뷰
- 우표 크게 보기 (톱니 프레임 유지) + 날짜 + 메모
- 액션: 메모 수정, 사진 교체, 삭제 (삭제는 확인 다이얼로그 필수)

### 3.4 리마인더 알림
- expo-notifications 로컬 알림 사용 (서버 푸시 불필요)
- 설정 화면에서 on/off + 시간 선택 (기본 21:00)
- 오늘 스탬프를 이미 붙였으면 그날 알림은 스킵 (알림 스케줄링 시 조건 처리)
- 알림 문구 예: "오늘의 우표를 붙일 시간이에요 📮"

### 3.5 설정 화면 (최소)
- 알림 on/off 및 시간
- 데이터 관련: 전체 스탬프 개수 표시, (v2 대비) "백업" 메뉴 자리만 placeholder
- 앱 정보/버전

## 4. 기술 스택

```
Expo SDK (최신 안정 버전, EAS Build 사용)
├── expo-router              # 파일 기반 라우팅
├── expo-image-picker        # 카메라/앨범
├── expo-image-manipulator   # 크롭/리사이즈/압축
├── expo-image               # 이미지 렌더링 (캐싱)
├── react-native-svg         # 톱니 우표 프레임 (clipPath 마스킹)
├── expo-sqlite              # 메타데이터 저장
├── expo-file-system         # 이미지 파일 저장
├── expo-notifications       # 로컬 리마인더
└── react-native-reanimated  # 붙이기 애니메이션
```

상태 관리는 가볍게 (zustand 권장). 서버 없음 — v1은 순수 클라이언트 앱.

## 5. 데이터 설계 (v2 동기화 대비 필수)

### SQLite 스키마
```sql
CREATE TABLE stamps (
  id TEXT PRIMARY KEY,          -- uuid (서버 동기화 대비, auto-increment 금지)
  date TEXT NOT NULL UNIQUE,    -- 'YYYY-MM-DD', 하루 1장 제약
  image_path TEXT NOT NULL,     -- 압축 원본 (장변 1200px)
  thumb_path TEXT NOT NULL,     -- 그리드용 썸네일 (장변 300px)
  memo TEXT,
  created_at TEXT NOT NULL,     -- ISO 8601
  updated_at TEXT NOT NULL,     -- ISO 8601 (동기화 충돌 해결용)
  deleted_at TEXT               -- soft delete (동기화 대비)
);
```

### 이미지 저장
- 위치: FileSystem.documentDirectory + 'stamps/'
- 저장 시 즉시 2벌 생성: 원본 압축본(1200px, JPEG q0.8) + 썸네일(300px, q0.7)
- 파일명: `{uuid}_full.jpg`, `{uuid}_thumb.jpg`

### 확장성 원칙 (중요)
- 데이터 접근은 repository 계층으로 추상화할 것 (StampRepository 인터페이스 → 지금은 LocalStampRepository 구현체만). v2에서 SyncedStampRepository로 교체 가능하게
- id는 uuid, 타임스탬프는 ISO 8601, soft delete — 전부 서버 동기화 대비
- 이미지 경로는 절대경로 저장 금지 (iOS는 앱 업데이트 시 컨테이너 경로 변경됨). 상대경로만 저장하고 런타임에 documentDirectory와 조합

## 6. 톱니 우표 프레임 컴포넌트 (핵심 정체성)

- `<StampFrame width height>` 재사용 컴포넌트로 구현
- react-native-svg의 Path + ClipPath 사용: 사각형 둘레에 작은 반원(perforation)을 일정 간격으로 파낸 path를 수학적으로 생성
- 파라미터: 톱니 반지름(칸 크기에 비례), 톱니 간격, 테두리 여백
- 빈 칸 상태와 사진 채워진 상태 둘 다 이 컴포넌트 하나로 처리
- 그리드(작은 크기)와 상세 뷰(큰 크기) 모두에서 톱니가 뭉개지지 않아야 함

## 7. 디자인 방향

- 레퍼런스: 하늘색(#A8D8EA 계열) 배경 + 회색빛 흰색(#F0F0F0) 빈 우표 칸
- 전체적으로 파스텔, 여백 넉넉, 장식 최소
- 폰트: 제목은 세리프 계열(레퍼런스 노트 표지 느낌), 본문은 시스템 기본
- 다크모드: v1 제외 (라이트 고정)
- 언어: 한국어 우선, 문자열은 상수 파일로 분리 (추후 i18n 대비)

## 8. 개발 순서 제안 (마일스톤)

1. **M1 — 뼈대**: Expo 프로젝트 셋업, 라우팅, StampFrame 컴포넌트 완성 (톱니 퀄리티가 앱의 생명이므로 최우선)
2. **M2 — 코어 루프**: 월별 그리드 + 사진→크롭→붙이기 플로우 + SQLite/파일 저장
3. **M3 — 완성도**: 상세 뷰, 수정/삭제, 붙이기 애니메이션, 과거 날짜 채우기
4. **M4 — 마무리**: 리마인더 알림, 설정 화면, 빈 상태(첫 실행) 온보딩, 아이콘/스플래시
5. **M5 — 배포 준비**: EAS Build, TestFlight/내부 테스트 배포

## 9. 완료 기준 (Definition of Done)

- 설치 → 로그인 없이 즉시 오늘 스탬프 생성 가능
- 앱 재시작/업데이트 후에도 데이터 유지 (경로 이슈 없음)
- 한 달치(31개) 스탬프가 채워진 그리드가 60fps로 부드럽게 스크롤
- 알림이 설정 시간에 정확히 오고, 이미 기록한 날은 오지 않음
- 하루 1장 제약이 UI와 DB 양쪽에서 보장됨

## 10. 열린 질문 (발주자가 추후 확정)

- 앱 이름 / 아이콘 디자인
- 온보딩 화면 유무와 문구
- 사진 교체 시 히스토리 보관 여부 (v1은 덮어쓰기로 가정)
- 월 그리드 열 수(4열 vs 5열)는 프로토타입 보고 결정
